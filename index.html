从零构建 AI Agent 与 Harness Engineering 实战课程

一门面向工程师的动手课：先手写一个 Agent，再把它打磨成可靠的运行时（Harness）。本课程以 Harness 12-Factor 为主线，每章对应一条生产级设计原则。

不要把 Agent 当 AI 来学，把它当成一个不可靠的分布式系统来学。

前置要求：Python 基础、命令行、一个 LLM API Key（推荐 Anthropic Claude，亦可用兼容 API）。

前置：环境与第一次 API 调用

学习目标：搭好环境，理解一次 LLM 请求/响应最小结构。

核心内容

  • SDK 安装、API Key 环境变量配置。
  • 最小调用三要素：system prompt、messages、模型参数。
  • 理解 token 计费与长度约束。
  • 流式与非流式响应差异。

动手练习

  1. 写 chat.py，命令行输入一句话并打印回复。
  2. 加多轮循环并维护 messages，观察 context 增长。
产出物：可多轮对话并打印 token 消耗的命令行脚本。

Factor 1：Single Agent Loop

学习目标：实现带工具的 REPL 循环，理解 Agent 本质是状态机。

核心内容

  • 调模型 → 解析工具调用 → 执行工具 → 结果回填 context 直到完成。
  • Tool calling 协议（name/description/参数 schema）。
  • 解析 tool_use 并映射到 Python 函数。
  • 模型不再请求工具时终止循环。

动手练习

  1. 不用框架，约 100 行实现 loop。
  2. 注册第一个工具（如 run_command / 文件读取 / 计算器）。
  3. 完成 2-3 步工具调用任务并打印每一步动作。
产出物：最小但完整的多步 Agent。

Factor 2：Explicit Tool Contract

学习目标：设计让模型用得对、用得稳的工具接口。

核心内容

  • 工具 description 本身就是 prompt。
  • 参数校验必须在工具内进行。
  • 返回值要利于模型继续决策。
  • 错误作为正常返回值，避免 loop 崩溃。
  • 有副作用工具要考虑幂等。

动手练习

  1. 重构 Factor 1 工具，加参数校验和友好错误。
  2. 设计有副作用工具并防止重复执行。
  3. 故意写模糊 description，观察误用。
产出物：一套健壮工具规范 + 2-3 个示例工具。

Factor 3：Context Budgeting

学习目标：处理 context 爆掉问题。

核心内容

  • 长任务和大工具输出会快速撑满窗口。
  • 策略：截断、摘要、外置记忆。
  • 在信息损失和成本之间做权衡。
  • 区分常驻 system prompt 与动态注入信息。

动手练习

  1. 用超长输出工具复现 context 爆掉。
  2. 实现截断与阈值摘要策略。
  3. 比较两种策略对任务质量影响。
产出物：长任务下仍稳定的 Agent。

Factor 4：Failure-First Design ★

学习目标：系统性修复死循环、跑偏、幻觉调用与部分失败。

核心内容

  • 死循环检测与打断（最大步数、重复检测）。
  • 阶段目标与自检避免跑偏。
  • 拦截未知工具与异常参数。
  • 多步任务的部分失败恢复。
  • 限流/超时重试与退避。

动手练习

  1. 人为注入工具错误、限流、未知工具。
  2. 为每种失败加 Harness 防护。
  3. 记录真实任务跑偏位置并改进。
产出物：能从多种错误中恢复的 Agent。

Factor 5：Graceful Degradation

学习目标：模型或工具失败时，系统降级而不是崩溃。

核心内容

  • 降级策略金字塔：完全成功 → 部分成功 → 安全兜底 → 优雅失败。
  • 模型乱答/拒绝时用更简单 prompt 或预设模板重试。
  • 工具不可用时报错并让模型换路径。
  • 多步任务中保留已完成部分。
  • 给用户清晰状态说明，而非堆栈。

动手练习

  1. 给 Factor 4 的 Agent 加"降级模式"开关。
  2. 模拟工具完全不可用，观察 Agent 给出替代方案。
  3. 设计连续乱答时的安全兜底回复。
产出物：失败时 degrade 而非 crash 的 Agent。

Factor 6：Least-Privilege Tooling

学习目标：让 Agent 安全地访问真实世界。

核心内容

  • 工具白名单与上下文级权限控制。
  • 沙箱执行：文件系统和网络隔离。
  • 最小权限原则：默认只读，显式授权写入。
  • 危险操作分类：自动放行 / 必须确认 / 永远禁止。

动手练习

  1. 实现只读模式拦截写操作。
  2. 给副作用工具增加确认门。
  3. 编写权限模型清单。
产出物：默认最小权限、危险操作需确认的 Agent。

Factor 7：Human-in-the-Loop Gates

学习目标：不可逆操作暂停等人确认，状态可恢复。

核心内容

  • 确认门状态机：运行中 → 挂起等待 → 恢复执行 → 完成。
  • 挂起时保存完整 context 与 trace，支持退出后恢复。
  • 批量确认 vs 单条确认。
  • 超时策略与审计日志。

动手练习

  1. 把确认做成 loop 内显式状态。
  2. 实现挂起-恢复：程序退出后重载状态继续执行。
  3. 给确认门加超时与审计事件。
产出物：具备可恢复人工确认门的 Agent。

Factor 8：Observable by Default

学习目标：让每次运行可见、可回放、可 debug。

核心内容

  • 结构化事件：思考、工具调用、结果、错误、完成。
  • 结构化日志：输入输出、token、耗时。
  • 基于 trace 复现与排障。
  • 成本观测：token、时延、费用。

动手练习

  1. 输出 JSONL 事件日志。
  2. 实现一个简易 trace 时间线查看器。
  3. 用 trace 定位一次跑偏根因。
产出物：每次运行都可追踪回放的 Agent。

Factor 9：Reproducible Runs

学习目标：给定 trace 和配置，能复现或定位一次失败运行。

核心内容

  • 复现三要素：trace + config + seed。
  • loop 路径与工具调用可复现，LLM 输出本身有随机性。
  • 把失败 trace 变成回归测试。
  • replay 模式：用旧 trace 替代真实调用，快速验证 Harness。

动手练习

  1. 保存一次失败运行的 trace + config。
  2. 实现 replay 模式，按记录推进 loop。
  3. 写测试：给定失败 trace，Agent 必须在第 N 步触发降级。
产出物：失败可被复现、修复可被测试验证的 Agent。

Factor 10：Composable Agents

学习目标：从单 Agent 升级到主-子编排器。

核心内容

  • 适用场景：并行分解、上下文隔离、角色分工。
  • orchestrator-worker 模式：拆分、派发、汇总。
  • 并行执行与结果合并策略。
  • 子 Agent 失败隔离与恢复。

动手练习

  1. 实现主 Agent 拆分子任务。
  2. 并行运行多个子 Agent 并汇总结果。
  3. 复用 Factor 8 trace 支持编排调试。
产出物：可并行执行并汇总的多 Agent 编排器。

Factor 11：Config-Driven Behavior

学习目标：把模型、prompt、工具列表、安全策略外置到配置。

核心内容

  • 什么该进配置：模型名、温度、max_tokens、system prompt、工具列表、安全策略等。
  • 按环境拆分配置：dev / staging / prod。
  • 用不同 config 做 A/B 测试。
  • 配置变更应触发自动化测试。

动手练习

  1. 把前面章节的硬编码参数抽到 config.yaml。
  2. 写"激进模式"与"保守模式"两种配置。
  3. 同一任务分别跑两种配置，观察行为差异。
产出物：行为完全由配置驱动的 Harness。

Factor 12：Continuous Evaluation

学习目标：用指标持续衡量 Harness 质量，而不是只看 demo。

核心内容

  • 核心指标：完成率、失败率、恢复率、平均步数/token/延迟/成本、人工介入率。
  • 评估数据集：收集真实任务和已知失败 case。
  • 批量 runner 自动判断成功/失败，输出报表。
  • 从指标回推设计：哪个 Factor 差就优先改进哪个。

动手练习

  1. 准备 10-20 个代表性任务（含陷阱 case）。
  2. 写 evaluate.py 批量运行并记录指标。
  3. 修改 Harness 后重跑评估，看关键指标是否提升。
产出物：能量化评估并持续改进的 Harness。

进阶：阅读真实 Harness

学习目标：带着问题读成熟 Harness 源码并迁移改进。

核心内容

  • 重点看 context 管理、容错、权限、可观测、编排、配置、评估。
  • 按问题驱动阅读，不盲目通读。
  • 抽取可迁移设计并落地到自己的 Agent。

动手练习

  1. 围绕 6 个问题各写一段精读笔记。
  2. 迁移至少 1 处设计到自己的 Agent。
产出物：源码精读笔记 + 至少一项工程改进。

结课项目：一个完整的领域 Agent

目标：综合所有 Factor，交付一个真实可用的领域 Agent。
  • 运维/网络方向：并行排查多设备，写操作需确认，全程 trace。
  • 数据方向：自主查询、清洗、生成报告。
  • 开发方向：读代码、定位 bug、提出修复。
